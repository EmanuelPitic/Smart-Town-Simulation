## 4. Broker

An MQTT broker acts as a communication bridge between client devices and plays a central role in the protocol's operation. In the MQTT v5 version, the broker received enhancements to provide additional functionalities and better meet current needs.

### 1. Connection Management
The broker receives connections from clients, which initially send a **CONNECT** packet. In version v5, the CONNECT packet includes additional properties, such as the Session Expiry Interval or user-specific properties.

### 2. Subscription Management
The broker keeps a record of all subscriptions. When a client wishes to subscribe to a topic, it sends a **SUBSCRIBE** packet, and the broker responds with a **SUBACK**. In MQTT v5, the SUBACK can include clear reason codes for the success or failure of the subscription process.

### 3. Message Publishing and Routing
When a client publishes a message, the broker determines the recipients based on existing subscriptions. **PUBLISH** packets in MQTT v5 can contain a "Topic Alias" to minimize packet size when long topics are frequently used.

### 4. Quality of Service (QoS) Management
The broker ensures message delivery according to the requested QoS level, which can be:
* **QoS 0**: At most once
* **QoS 1**: At least once
* **QoS 2**: Exactly once

### 5. Advanced Features in MQTT v5
* **Shared Subscriptions**: Allows multiple clients to subscribe to the same topic, with messages being distributed in a balanced way among group members.
* **Flow Control**: The client can specify how many messages it can receive before sending an acknowledgment.
* **Additional Properties**: Adds metadata to packets, such as reason strings for certain actions or message expiry intervals.

### 6. Security and Authentication
Most MQTT v5 brokers offer authentication methods via username/password, client certificates, and more advanced methods such as OAuth or JWT authentication.

## Topics

In the MQTT protocol, **topics** represent the channels through which messages are sent and received, playing an essential role in organizing and managing communication between devices.



### 1. Structure
* A topic is a string that defines the message path and is divided into several levels, separated by `/`. For example, `home/living_room/temperature` can represent a message sent from a temperature sensor in the living room.
* The topic hierarchy is defined by each level. In the previous example, `home` is the main category, `living_room` is a sub-level specifying location, and `temperature` defines the type of data transmitted.

### 2. Subscription and Publishing
* Clients "subscribe" to a topic to receive messages. Every time another client publishes a message to that topic, all subscribers will automatically receive it.
* It is possible to subscribe to multiple topics simultaneously, and for complex or unknown topics, wildcards can be used. For example, `home/living_room/#` will capture all messages under `home/living_room`, regardless of what follows.

### 3. Wildcards
* **The # (hash) wildcard**: Allows capturing all messages within a hierarchy. For example, `home/#` will include all topics under home, such as `home/living_room` or `home/kitchen/temperature`.
* **The + (plus) wildcard**: Allows subscribing to a single level of the hierarchy. For example, `home/+/temperature` will capture all temperature-type messages from any room in the house.

### 4. Security and Access
To maintain the security and integrity of messages, the MQTT broker can enforce strict rules regarding who can publish to or subscribe to certain topics. These controls are essential in industrial applications or sensitive IoT networks where data access must be restricted.

### 5. Retained Messages
If a message is published to a topic with the "retain" flag active, the broker will store that message as the most recent one sent on that topic. Thus, a new client subscribing to that topic will immediately receive the retained message, giving them access to the last available data even if it was published previously.

### 6. Message Persistence
Retained messages can be used to send important information, such as the status of a device or system. In situations where a new client connects for the first time or reconnects after a disconnect, the broker will automatically send the retained message, ensuring data continuity.

### 7. Message Handling
The broker manages the flow of messages according to client subscriptions. It decides which clients should receive published messages based on the topics they are subscribed to. This ensures efficient and organized communication without overloading the network with unnecessary messages.
